# Промпт-инжиниринг для разработчиков

> Часть [[Руководство разработчика]]

## Принцип: промпт — это ТЗ для агента

Чем точнее ТЗ, тем лучше результат. Разница между «напиши сервис» и хорошим промптом — как между «сделай приложение» и нормальной спекой.

## Анатомия хорошего промпта

```
[РОЛЬ]       Необязательно, но помогает для сложных задач
[КОНТЕКСТ]   Что агент должен знать о задаче
[ЗАДАЧА]     Конкретно, что нужно сделать
[ФОРМАТ]     Как должен выглядеть результат
[ОГРАНИЧЕНИЯ] Чего делать нельзя
```

### Пример: полный промпт

```
Контекст: проект на текущем стеке. На странице каталога растений
нужно добавить серверные фильтры и синхронизацию с URL.

Задача: реализуй фильтры по уровню освещения и сложности ухода.

Требования:
1. Backend:
   - FormRequest для query-параметров
   - фильтрация в query builder/scopes
   - передача filter state в Inertia::render
2. Frontend:
   - controls в resources/js/Pages/Plants/Index.vue
   - синхронизация с URL/query
   - empty state + loading state
3. Tailwind:
   - используй существующие классы/паттерны проекта
   - поддержи dark mode если есть на странице

Ограничения:
- Не добавляй новые зависимости
- Не меняй внешний API маршрута без явного согласования
- Не дублируй логику между страницами, вынеси общие части в composable

Начни с плана, потом реализация.
```

## Шаблоны промптов для типичных задач

### Новый endpoint

```
Создай endpoint [METHOD] [PATH] в [HANDLER_FILE]:

Бизнес-логика:
- [что должен делать]

Модель запроса: [описание полей]
Модель ответа: [описание полей]

Валидация: [какие проверки]
Авторизация: [какие права нужны]

Следуй паттерну [СУЩЕСТВУЮЩЕГО_КОНТРОЛЛЕРА].
Используй FormRequest и Policy где это уместно.
```

### Баг-фикс

```
Баг: [описание проблемы]
Воспроизведение: [шаги]
Ожидаемое поведение: [что должно быть]
Текущее поведение: [что происходит]
Файл: [путь]

Найди причину и предложи исправление. 
Добавь тест, который ловит этот баг.
```

### Рефакторинг

```
Файл/пакет: [путь]
Проблема: [что не так — дублирование, сложность, нарушение SRP]
Цель: [как должно стать]

Ограничения:
- Не меняй публичное API пакета
- Обеспечь обратную совместимость
- Все существующие тесты должны проходить

Покажи план изменений перед началом.
```

### Миграция БД

```
Создай миграцию для [описание изменения]:

Таблица: [имя]
Изменения: [добавить столбец / индекс / constraint]
Тип данных: [точный тип]

Требования:
- Миграция должна быть обратимой (up + down)
- Не теряй существующие атрибуты столбца при modify
- Используй стандартный формат Laravel migration class
```

### Генерация тестов

```
Напиши табличные тесты для [ФАЙЛ]:

Функция/метод: [имя]

Кейсы:
- Happy path: [описание]
- Ошибка валидации: [описание]
- Not found: [описание]
- Authorization denied: [описание]
- Edge cases: [нулевые значения, пустые строки, максимальные значения]

Следуй текущему стилю PHPUnit-тестов проекта.
```

## Приёмы для улучшения результата

### «Покажи план»
Перед крупными изменениями: `Покажи план перед тем как начнёшь менять файлы.`
Это позволяет скорректировать подход до изменений.

### «Следуй паттерну»
`Следуй паттерну [конкретный файл]` — агент скопирует стиль, структуру и конвенции из указанного файла. Мощнее любого описания стиля.

### «Объясни перед реализацией»
`Объясни свой подход и получи моё подтверждение перед тем как писать код.`
Для архитектурных решений.

### «Только X, больше ничего»
`Измени только функцию ParseOrder, больше ничего не трогай.`
Ограничивает scope изменений.

### Отрицательные примеры
```
НЕ делай так:
```php
// плохо
$user = User::query()->whereRaw("email = '{$email}'")->first();
```

Делай так:
```php
// хорошо
$user = User::query()->where('email', $email)->first();
```
```

### Цепочка промптов (chain of thought)
Для сложных задач разбивай на шаги:
```
Шаг 1: Проанализируй текущую реализацию в [файл]
Шаг 2: Предложи три варианта решения с trade-offs
Шаг 3: После моего выбора — реализуй
Шаг 4: Напиши тесты
Шаг 5: Запусти релевантные проверки проекта
```

## Антипаттерны промптов

| Антипаттерн | Почему плохо | Как исправить |
|---|---|---|
| «Сделай красиво» | Субъективно, нет критериев | Конкретные требования к стилю |
| «Оптимизируй» | Не ясно что оптимизировать | «Уменьши аллокации в hot path» |
| «Как обычно» | Агент не помнит «обычно» | Укажи файл-пример |
| Стена текста | Агент теряет главное | Структура с нумерацией |
| Промпт на 2 экрана | Слишком много за раз | Разбей на цепочку |
